LGTM · PR Review Assistant 架构

GitHub PR 评审助手 — Next.js 前端 + Go 后端 · 分层上下文 (L1–L4) + RAG · LLM 三阶段并发 SSE 流式评审 · GitHub App 自动评审

评审主流程（手动）

  • • ① 前端 POST /api/review 提交 PR URL
  • • ② 拉取 meta / diff / CI checks / 约定文档
  • • ③ 按 owner/repo/pr/head_sha 命中缓存可直接回放
  • • ④ 构建 L1–L4 上下文，按预算裁剪
  • • ⑤ summary / risks / suggestions 三阶段并发
  • • ⑥ SSE 推 summary_delta · *_done · review_id 并写历史

分层上下文 · RAG · Agent

  • • L1 PR meta（永久保留）/ L2 diff hunk
  • • L3 约定文件（README/CONTRIBUTING/CLAUDE/AGENTS）
  • • L4 RAG：SQLite 向量 · cosine ≥0.35 · top-4 · scope 隔离
  • • 每阶段不同 query（risks 偏安全/并发/泄漏）
  • • Agent ReAct：PR 沙盒三件套 + search_repo
  • • 沙盒工具只读本 PR 改动集，逃逸即拒绝

部署 · 存储 · GitHub 集成

  • • 前端 Vercel（standalone）· /api/* rewrite 直连后端
  • • 后端 Fly.io 东京(nrt) · /data volume 挂 SQLite
  • • Store: SQLite / Postgres · Cache: Memory / Redis
  • • LLM: DeepSeek（OpenAI 兼容）· Embedding 独立配置
  • • GitHub App: OAuth 登录 + webhook 自动评审
  • • 建议回写 PR：comment → GraphQL apply 成 commit

分层上下文 L1–L4 · 取数 → 预算 → 拼 Prompt → LLM

PR 的四个上下文来源各自取数，经 48k token 预算裁剪后按模板拼成一个 prompt，发给 LLM 的三阶段（summary / risks / suggestions）。代码见 backend/internal/prctx/layered.go

四层各取什么

  • • L1 PR 元信息：标题/正文/每文件增删行
  • • L2 Diff：每个改动文件的 patch hunk
  • • L3 约定：README/CONTRIBUTING/CLAUDE/AGENTS
  • • L4 RAG：同仓相关代码 chunk（cosine top-4）

预算与压缩

  • • 总额 48k token（按字符/3 粗估）
  • • 分配约 3:4:1:2，L1 永不压缩
  • • L2 拿剩余、保底 1000 token
  • • 压缩顺序 L3 → L4 → L2 → L1
  • • 超预算文件入 Dropped，前端标注未覆盖

拼接与使用

  • • 四层渲染进 *.tmpl 成一个 prompt
  • • 模板真实顺序：L1 → L3 → L4 → L2
  • • summary/risks/suggestions 共享 L1/L2/L3
  • • L4 各阶段用不同 query 重算召回

GitHub Webhook 自动评审 · 时序

pull_request.opened / synchronize / reopened（或 PR 评论 /lgtm review）→ HMAC 校验 → 202 立即返回 → 后台 goroutine 评审 → 回贴 bot review + 站内通知
