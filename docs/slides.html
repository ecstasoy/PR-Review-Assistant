问题 · the problem

Code review 是瓶颈，
而多数 AI 评审"说正确的废话"。

  • 人工评审耗时，依赖资深同事的注意力，PR 一多就积压。
  • 直接把 diff 丢给 LLM → 泛泛而谈：「注意命名」「记得加测试」。
  • 模型不懂这个仓库的约定、历史、跨文件关系 → 漏掉真正的风险。

→ 不是模型不行，是它看到的太少。

1.5 分钟。建立痛点：评审慢 + AI 评审水。点出关键转折——不是模型不行，是上下文不够。这句直接引到下一页主张。
核心主张 · thesis

好评审取决于上下文，
而不是把 diff 丢给模型。

所以 LGTM 主要在解决一件事：喂给模型什么。上下文分层、按预算裁剪、按阶段检索。

30 秒。全场记忆锚点，慢一点、停顿一下。后面所有技术都挂在这句上。

整体架构 · architecture
1.5 分钟。一句话扫过：浏览器→Vercel 前端→Fly 后端（核心引擎在这）→外接 LLM/存储/缓存，GitHub 双向。强调"核心都在后端那块高亮的框"，下面四页拆它。完整图留 Q&A。
核心 ① · 分层上下文

把上下文分成四层，按预算裁剪。

L1
PR 元信息 — 标题/正文/作者/标签/分支/文件统计/CI checks　（永久保留）
L2
Diff — 每个改动文件的 patch hunk
L3
项目约定 — README/CONTRIBUTING/CLAUDE.md/AGENTS.md
L4
RAG 召回 — 同仓历史代码 chunk（离线索引 + 评审写入）

预算约 3 : 4 : 1 : 2　·　L1 永留，L2 保底 floor 不被挤空，超预算文件进 dropped 上报前端。

2 分钟，差异化核心。重点不是"分了四层"，而是"有预算地选"。类比：context window 不是越满越好，是把额度花在最相关的地方。dropped 上报体现透明。
核心 ① · 预算分配算法

每层一个额度，按固定顺序切。

limit = 48000 # 64K 输入 − 16K 输出预留 L1 = tokens(PR 元信息) # 全留；超 limit 直接报错 L3 = min(limit × 10%, limit − L1 − 1000) L4 = min(limit × 20%, limit − L1 − L3 − 1000) # RAG 关 → 0 L2 = max(limit − L1 − L3 − L4, 0) # 装不下 → Dropped

L1 最先占且永不压缩　·　1000 是 L2 的保底，谁都不能把它挤空　·　token 按 字符数/3 粗估

~1 分钟，并进核心①的预算讲解里。一句话：额度按固定顺序切——L1 先占且永不动，L3 拿 10%、L4 拿 20%，L2 拿剩下的但保底 1000，所以 L3/L4 再大也挤不空 L2；装不下的文件标 Dropped 告诉前端。token 是字符数/3 粗估，没引真分词器。
核心 ② · RAG 召回

RAG 召回按阶段定制 query。

  • scope = owner/repo：只在本仓库召回，绝不跨仓串数据。
  • 本 PR 已在 L2 的文件从 L4 跳过，避免重复占预算。
  • cosine ≥ 0.35 · top-4，低相关直接过滤。
  • 每个阶段不同 query：risks 偏 bug/安全/并发；suggestions 偏 重构/性能。
1 分钟。一句话立住差异：同一个 PR，风险阶段和建议阶段检索的东西不一样。这是"上下文随用途变"的具体体现。
核心 ③ · 三阶段并发 + 流式

三个阶段并发跑，摘要边写边出。

  • summary · risks · suggestions 并发，互不阻塞。
  • 摘要走流式 markdown（体感快）；风险/建议走 JSON（response_format）。
  • SSE 事件：summary_delta · risks_done · suggestions_done · budget_report。
  • 按 head_sha 缓存：同一 commit 直接回放历史结果。
1 分钟。承接 Demo 里看到的流式效果——这页解释它怎么来的。head_sha 缓存是省钱省时的工程细节，评委爱听。
核心 ④ · Agent 追问

评审完还能追问，但 agent 有边界。

  • ReAct loop：LLM → 调工具 → 结果回灌 → 再调，直到答完。
  • PR 沙盒：read_file / list_dir / grep_patches —— 只读本 PR 改动集。
  • RAG 工具：search_repo 在仓库 scope 内语义检索。
  • 逃出 PR 改动集直接拒绝，不碰本地文件系统。
1 分钟。卖点是"有边界的 agent"——安全是设计进去的，不是事后补。对评委这是成熟度信号。

工程实现 · engineering

存储、缓存、模型都能换，失败优先降级。

可替换

  • 多供应商模型注册表 + 按阶段 / 按请求路由
  • 存储 SQLite ↔ Postgres
  • 缓存 Memory ↔ Redis（跨实例共享）

稳健

  • 降级优先：缺 key / store 失败 → 显式 warn + 降级
  • 单点失败不拖垮整服务，nil-safe handler

可观测

  • OpenTelemetry traces · Sentry · slog JSON
  • liveness / readiness 双健康检查

部署

  • Fly.io 后端 + Vercel 前端
  • SSE 浏览器直连后端，绕开边缘函数超时
1 分钟。给评委"工程成熟度"信号。别逐条念，挑两点：模型注册表（最近三个 PR 在做）+ 降级优先。其余扫过。
边界 · known limits

现在的几个限制。

  • ListFiles 当前只取第一页（≤ 100 文件），超大 PR 会丢后续文件。
  • L2 当前是 patch hunk，FileContext.FullText 字段已预留未接。
  • RAG 是文本 chunk + cosine，还不是真实引用关系。
45 秒。反直觉但加分：评委看几十个项目，主动讲限制的最可信。语气从容，不是认错，是路线图入口。
未来方向 · roadmap

接下来想做的几件事。

  • tree-sitter / LSP / 调用图 → 把 RAG 升级成真实的符号引用。
  • 异步索引 + 队列（worker · Redis Streams · job table），评审不被索引拖慢。
  • 向量存储升级：pgvector / Qdrant，撑万级以上 chunk。
  • 评测 harness：误报 / 漏报 / 可应用率 / 成本 —— 用数据驱动模型选择。
45 秒。挑评测 harness 重点讲——没有评测就没法判断换模型是否真变好，这点最能体现你想清楚了。
